Escala y costos

¿Cómo escala el sistema a millones de comprobantes?

El cuello de botella es Postgres, y con los índices apropiados (B-tree para hashes, GIN para trigramas, IVFFlat para vectores) maneja millones de filas sin esfuerzo. A 1M de clientes con ~5 reglas promedio = ~5M filas, lookups en microsegundos. Si supera ~10M filas, particionamos por client_cuit.

Y lo más importante: el % de uso de LLM decrece con el tiempo porque más reglas compiladas = más matches deterministas. La curva de costo es invertida — a más tráfico, más barato por unidad.

¿Cuánto cuesta el sistema a régimen?

A 100 clientes ~USD 35/mes. A 1K clientes ~USD 55/mes. A 100K clientes ~USD 250/mes. A 1M clientes ~USD 450/mes.

Pasar de 100 a 1M de clientes (10.000× más volumen) solo multiplica el costo por ~13×. La mayor parte es Supabase (Postgres + pgvector); la API de Anthropic Haiku tiende a usarse menos con el tiempo.

¿Qué pasa si la API de IA (Anthropic / OpenAI) se cae?

El ~95% del sistema sigue funcionando. Los 3 niveles de la cascada son Postgres puro, sin dependencia online del LLM. Solo se degradan dos cosas: (a) la inferencia con DNA para casos genuinamente nuevos, y (b) la compilación de respuestas nuevas (la queue del Learning Pipeline acumula tareas y se procesan cuando vuelve la API).

En un RAG, en cambio, un outage del LLM detiene 100% del tráfico.

Funcionamiento concreto

Cliente que siempre compra sillas negras, pero un día el comprobante dice "sillas con patas". ¿Cómo se resuelve sin molestar al cliente?

Asumiendo que es una observación de facturación (no la descripción del producto):

  1. Llega el comprobante → lookup del Client DNA por CUIT → su quirks_digest dice algo como "siempre 50/50 entre dos razones sociales".
  2. La cascada corre: exact falla, fuzzy falla (no se parece a "sillas negras"), semantic falla ("patas" no mapea a concepts conocidos).
  3. Inferencia con DNA: el sistema le pasa a Haiku la observación + el digest del cliente. Si infiere con confianza, ejecuta. Si no...
  4. Escalación al operador interno (NUNCA al cliente). El operador responde: "misma división de siempre".
  5. El Learning Pipeline compila esa respuesta como regla per-client. Próxima vez: exact match, <1ms, sin operador.

El cliente envió un comprobante normal y recibió su confirmación. Nunca fue molestado.

¿Y si después aparece "sillas con patas plásticas" y el sistema lo confunde con la regla nueva?

Acá entra el aprendizaje negativo (§12). El operador corrige el match incorrecto. El sistema automáticamente:

  • Guarda "sillas con patas plásticas" como contra-ejemplo de la regla "sillas con patas" (no corrige la vieja, marca su frontera).
  • Compila una regla nueva para "sillas con patas plásticas" con la acción correcta.

Próxima vez que el matcher considere matchear contra "sillas con patas", chequea los contra-ejemplos: si la observación es más cercana a un contra-ejemplo que al trigger positivo, rechaza el match y baja al siguiente nivel. Cada corrección del operador entrega valor en dos dimensiones: regla nueva + frontera de regla vieja.

¿Qué pasa si un cliente cambia su patrón con el tiempo (antes pedía 50/50, ahora 60/40)?

Cuando el operador corrige el match: la regla vieja se desactiva (soft delete, no se borra para mantener audit trail) y se compila una regla nueva con la acción correcta. El quirks_digest del Client DNA se recompila para reflejar el cambio.

Próximos comprobantes matchean la regla actualizada. El historial de reglas inactivas queda para auditoría pero no participa del matching.

Por qué esta propuesta y no otra

¿No es esto un rules engine clásico (tipo Drools) con un loop de aprendizaje?

La similitud existe — un rules engine también matchea reglas y ejecuta acciones. Tres diferencias:

  1. Las reglas se compilan automáticamente desde lenguaje natural, no las escribe un developer.
  2. La cascada de 3 niveles ordenada por costo (exact → fuzzy → semantic) es atípica en rules engines clásicos. Captura variaciones lingüísticas sin pagar el costo del LLM en cada query.
  3. Los dos diferenciales §11 (pre-compilación predictiva) y §12 (aprendizaje negativo) son ortogonales al patrón clásico — son lo que un prompt genérico no propondría.

Honestamente: el corazón es un rules engine + memory system. La novedad está en las extensiones específicas al dominio de Galo. La nota honesta del README lo reconoce abiertamente.

¿Es solo RAG con otro nombre?

No. RAG guarda texto y le pide al LLM que lo interprete en cada query. Compiled Intelligence guarda reglas estructuradas y las ejecuta sin LLM en el 95% de los casos.

La diferencia operativa: RAG paga el costo de pensar cada vez. Este sistema piensa una vez (al compilar) y ejecuta siempre (al matchear). Diferentes filosofías → diferentes curvas de costo, latencia, y tolerancia a fallos.

¿Por qué Postgres y no una vector DB dedicada (Pinecone, Weaviate)?

Postgres + pg_trgm + pgvector cubren todo en un solo sistema (relacional + texto + vectores). Una vector DB dedicada agrega red, latencia y operaciones extras sin beneficio claro hasta volúmenes muy grandes. USD 25/mes de Supabase Pro alcanza hasta 10K clientes.

¿Por qué Haiku y no Sonnet u Opus?

Las tareas que la IA hace son simples: clasificar binario (per-client vs global), extraer JSON con schema cerrado, resumir reglas en ~150 tokens. Sonnet/Opus serían over-engineering: 20-60× más caros, latencia mayor, sin beneficio medible.

Si en algún path puntual hace falta más potencia, se sube solo ese path. No hay razón para pagar Opus para clasificar "per_client" vs "global".

Datos, errores y producción

¿Qué pasa si el operador responde mal una pregunta del bot?

Mitigaciones en capas:

  • Las reglas recién compiladas arrancan con confidence baja → el sistema pide confirmación antes de ejecutar autónomamente.
  • Cualquier corrección posterior desactiva la regla mala y compila una nueva (más el contra-ejemplo §12).
  • Las reglas con muchas correcciones se marcan como volátiles y se priorizan para revisión humana.
¿Qué pasa si Haiku misclasifica una regla per-client como global (o viceversa)?

Acá entra la cuarentena de promoción global (§5 + §6.7): el sistema no auto-promueve a global cuando 3 clientes coinciden. Pasa por un estado intermedio candidate_global que no se aplica autónomamente a otros clientes — solo se le propone al operador como sugerencia cuando viene un comprobante relevante. Después de ~5 validaciones humanas, pasa a global activa.

Esto evita el envenenamiento progresivo que vendría de una cadena de misclasificaciones.

¿Cómo se borra el historial de un cliente que se va?

Soft delete con audit trail. Las reglas y el Client DNA se marcan active = false, no se purgan inmediatamente — por compliance y por si el cliente vuelve. Un job de retención puede purgarlos definitivamente después de N meses si la política lo requiere.

¿Cómo migraríamos a producción desde este diseño?

Rollout sugerido en 3 fases:

  1. Semana 1-2: levantar Postgres + extensiones + schema. Cargar Client DNA basado en lo que Galo ya tiene. Correr el job de pre-compilación predictiva (§11) sobre el historial de pedidos.
  2. Semana 3-4: dual-write: el bot existente sigue funcionando, pero también consulta el Rules Engine en paralelo y compara. Sin impacto en producción real.
  3. Semana 5+: activar el Rules Engine como path principal con escalación al operador para casos sin match. Métricas: resolution_rate, % de correcciones, tiempo medio del operador por intervención.

Nota honesta del autor

La arquitectura central de esta submission — "compilar respuestas humanas en reglas reutilizables" — es la respuesta esperable que cualquier IA moderna le da a este prompt. Para diferenciarnos, sumamos tres ideas que un prompt genérico no propone porque requieren leer el contexto específico de Galo. Ver la sección §7. Diferenciales al final de este panel.

1¿Qué problema resuelve?

En una línea: los clientes mandan comprobantes por WhatsApp con notas ambiguas. Si el bot pregunta dos veces lo mismo al mismo cliente, se ofende. El sistema tiene que aprender una vez y nunca volver a preguntar.

Galo procesa miles de comprobantes por día. La parte mecánica (extraer monto, CUIT, banco de la imagen) ya está resuelta. El problema son las observaciones libres tipo «armar factura A y B» o «hacer 50/50» — donde el bot necesita interpretar qué quiere el cliente.

Lo crítico: el cliente ya envió esta misma nota la semana pasada. Si volvemos a preguntarle qué significa, queda mal. Necesitamos memoria — pero una que sea barata, rápida y escale a millones de comprobantes.

2La idea: compilar, no interpretar

En una línea: en vez de guardar las aclaraciones como texto para que la IA las relea cada vez, las traducimos a reglas determinísticas que se ejecutan sin IA.

Pensalo como la diferencia entre un intérprete y un compilador:

  • Intérprete (RAG clásico): cada vez que llega un comprobante, la IA lee la observación, busca textos parecidos en una base, los lee, decide. Lento, caro, no determinístico.
  • Compilador (Compiled Intelligence): la primera vez la IA compila la respuesta del operador en una regla con trigger y acción. Las siguientes veces, un lookup en Postgres devuelve la regla en milisegundos. Sin IA, gratis.

RAG (intérprete)

  • cada query usa IA
  • la IA puede variar entre llamadas
  • $ crece con el uso
  • si la IA cae, todo cae
~$0.003-0.01 / query · ~500ms-2s

Compiled Intelligence

  • la IA aprende una vez
  • resultado siempre igual
  • $ baja con el uso
  • 95% sobrevive caídas de IA
$0 / query (95% casos) · <5ms
Mismo problema, dos filosofías opuestas
RAG resuelve el problema, pero re-paga el costo de pensar en cada query. Si el problema es estable (mismo cliente, misma nota), tiene más sentido pensar una vez y guardar el resultado en código.

3Los 3 niveles, en simple

En una línea: el sistema busca la respuesta en orden, de más barato a más caro. Se corta apenas encuentra match.

Cada comprobante con observación ambigua pasa por una cascada de 3 niveles. La cascada se diseñó para que el caso común sea casi gratis y el caso raro sea caro pero infrecuente:

Distribución típica de comprobantes a día 90 · cada nivel es independiente
  • Nivel 1 (Exact): ¿esta frase la vi exactamente igual antes? Es un lookup en una tabla. Tarda menos de 1 milisegundo.
  • Nivel 2 (Fuzzy): ¿la frase se parece a algo que vi, aunque tenga typos? Captura «fact. A & B» contra «factura A y B».
  • Nivel 3 (Semantic): ¿la frase significa lo mismo que algo que vi, aunque use otras palabras? Captura «dividir en dos partes iguales» contra «50/50».
  • Escalación: si nada matcheó, el bot le pregunta al operador humano (NUNCA al cliente). El operador responde una vez, y el Learning Pipeline compila la respuesta en regla para que nunca más haga falta preguntar.
Cada nivel cubre un tipo distinto de variación lingüística. Cortar en el primer match minimiza el costo: si exact resuelve, no pagamos fuzzy ni semantic.

5Per-client vs global

En una línea: una misma frase puede significar cosas distintas para clientes distintos. El sistema mantiene reglas específicas por cliente y reglas universales — y siempre prefiere la específica.

Para el cliente ABC, «50/50» significa «factura A y B en partes iguales». Para el cliente XYZ, significa «dividir el monto en dos comprobantes separados». Misma observación, distinta acción.

Si una regla per-client de 3+ clientes diferentes coincide en trigger y acción, el sistema la promueve a global automáticamente — pero las per-client se mantienen como overrides.

El default conservador es per-client (peor caso: redundancia). El sistema sube a global solo cuando hay evidencia de que aplica a muchos.

6Decisiones clave

¿Por qué Postgres y no una vector DB dedicada (Pinecone, Weaviate)?
Postgres + pg_trgm + pgvector cubren todo en un solo sistema (relacional + texto + vectores). Una vector DB dedicada agrega red, latencia y operaciones extras sin beneficio claro a esta escala.
Costo USD 25/mes (Supabase Pro) hasta 10K clientes.
¿Por qué Haiku y no Sonnet u Opus?
Las tareas que la IA hace acá son simples (clasificar binario, extraer JSON estructurado, resumir reglas). Haiku alcanza, es 20-60x más barato, y la latencia es menor.
Si en algún path puntual hace falta más potencia, se sube solo ese path.
¿Por qué no preguntarle al cliente por más contexto?
Porque el cliente es recurrente. La segunda repregunta lo ofende. Toda interacción se resuelve internamente con el operador, una sola vez.
Esto convierte a la memoria en un requerimiento de negocio, no en optimización.
¿Por qué tres niveles y no uno solo (todo semantic)?
Porque el 75%+ de los casos a día 90 son repeticiones exactas. Pagar el costo del embedding (USD 0.0001 + 100ms) en cada query, cuando un lookup en B-tree resuelve en <1ms gratis, es un desperdicio sistemático.
Cada nivel cubre un tipo de variación distinto, ordenado por costo creciente.
¿Qué pasa si Anthropic / OpenAI caen?
El 95% de los comprobantes sigue resolviéndose (los 3 niveles deterministas son Postgres puro). Solo se degradan: (a) los pocos casos completamente nuevos, y (b) la compilación de respuestas nuevas (la queue se acumula y se procesa cuando vuelve).
Contrast: en RAG, un outage del LLM detiene 100% del tráfico.
¿Y la facturación / logística?
No las hace Galo. Galo orquesta. La «acción» final del sistema es una instrucción estructurada que se transmite al ERP de cada empresa cliente. Esto simplifica el modelo: no son operaciones críticas, son mensajes pasables.
Limita el alcance de la submission a lo que efectivamente tiene que resolver Galo.

7Diferenciales sobre la solución genérica

En una línea: sabemos que la idea "compilar respuestas en reglas" la propone cualquier IA. Por eso agregamos estos tres ángulos que requieren leer el contexto específico de Galo, no solo el problema abstracto del challenge.
① Pre-compilación predictiva desde pedidos history
El sistema NO arranca vacío. Un job background mina el historial de pedidos que Galo ya tiene per-client y pre-popula reglas de baja confianza ANTES de que llegue el primer comprobante. Cuando llega ese primer comprobante, el operador valida con un click en lugar de explicar desde cero. Día 1 pasa de ~5-10% de cobertura a ~30-45%.
Una IA genérica piensa en el sistema de memoria aislado. No razona sobre datos laterales que la empresa ya tiene y que se pueden explotar para arrancar con cobertura. Detalle en ARCHITECTURE §11.
② Aprendizaje negativo / reglas de frontera
Cada corrección del operador genera tanto una regla nueva como un contra-ejemplo asociado a la regla que se equivocó. El matcher consulta esos contra-ejemplos: si una observación entrante es más cercana a un contra-ejemplo que al trigger positivo, se rechaza el match. Las reglas aprenden explícitamente sus límites.
La mayoría de propuestas solo guardan ejemplos positivos. Las correcciones humanas tienen valor compuesto cuando se aprovechan en ambas direcciones (qué es vs qué NO es). Detalle en ARCHITECTURE §12.
③ Demo + drawer pedagógico en un solo HTML
El evaluador prueba el sistema sin instalar nada. Doble click en demo.html, lee la explicación con diagramas (este panel), tira los ejemplos precargados.
La mayoría de submissions van a entregar solo doc o solo código. Acá ves el sistema "andando" antes de leer 500 líneas de markdown.
Este demo simula la lógica del Rules Engine y el Learning Pipeline 100% en el browser. La normalización, hash, similitud de trigramas y match semántico están implementados en vanilla JS sobre datos en memoria. El nivel semántico usa un mock honesto basado en concept_tags + un mapa de sinónimos (en producción es pgvector + text-embedding-3-small). No se hacen llamadas a APIs externas.
